Understanding the Internet — From Telephone Lines to the Web

By Champak Roy — learnwithchampak.live · Beginner guide · Illustrated

Before computers were networked, telephones gave people a way to communicate across distances. That old system helps us understand how the Internet evolved. This post starts with telephone networks and builds step-by-step to IP addresses, protocols, HTTP, Telnet and the client–server model — with simple animated illustrations to make the ideas stick.

1. From telephones to packet switching

Telephone networks originally used circuit switching: a dedicated line is opened between caller and receiver for the whole call. That wastes resources if nobody speaks. The Internet uses packet switching — messages are split into packets, each routed independently and reassembled at the destination.

Why packet switching?

  • Efficient — multiple conversations share the same link.
  • Resilient — packets can take alternate routes if a link fails.
  • Scalable — easier to add more devices.

2. IP addresses — the Internet's postal address

Every device on the network needs an address. The Internet Protocol (IP) gives each device a unique identifier so packets know where to go.

IPv4 example

192.168.1.10

IPv4 uses four numbers (0–255) separated by dots. There are only ~4.3 billion addresses in IPv4, which is why IPv6 was introduced:

IPv6 example

2400:cb00:2048:1::c629:d7a2

3. What is a protocol?

A protocol is a set of rules that both ends follow to exchange data. If two devices speak the same protocol, they understand each other.

Common protocols

  • HTTP / HTTPS — the web's protocol for transferring pages and resources.
  • DNS — translates domain names (like learnwithchampak.live) into IP addresses.
  • FTP — file transfer.
  • SMTP / IMAP / POP — email delivery & retrieval.
  • Telnet / SSH — remote shell access (Telnet is unencrypted; SSH is secure).

HTTP — simple request / response

When you open a web page, your browser sends an HTTP request and the server returns an HTTP response.

GET /index.html HTTP/1.1
Host: learnwithchampak.live

-- server responds --

HTTP/1.1 200 OK
Content-Type: text/html
<html>...page...</html>

HTTPS adds encryption (TLS) so the content and credentials are protected in transit.

4. Telnet — early remote access (text only)

Telnet was an early tool to log into remote machines and type commands. It sends text across the network unencrypted, so modern usage is discouraged in favor of SSH (Secure Shell).

Telnet example (historical / educational):

telnet towel.blinkenlights.nl
-- plays Star Wars in ASCII --

Use ssh user@server for secure remote login instead of telnet.

5. Client–Server model: the Internet’s main pattern

Most internet interactions use the client-server model: clients (browsers, apps) request resources; servers (web servers, databases) respond. The server can be a physical machine or a cloud-hosted service.

Example flow when you visit a website

  1. You type learnwithchampak.live.
  2. Your browser asks DNS for the site's IP address.
  3. Your browser opens a TCP connection to that IP (port 80 for HTTP, 443 for HTTPS).
  4. The browser sends an HTTP request and the server sends back HTML/CSS/JS and media.
  5. Your browser renders the page.

6. Putting it all together — short glossary

Packet
Small chunk of data with header (address info) and payload (actual data).
IP Address
Numeric address assigned to a device (IPv4 or IPv6).
Protocol
Rules for communication (HTTP, DNS, TCP, UDP, SSH).
Router
Network device that forwards packets between networks.
